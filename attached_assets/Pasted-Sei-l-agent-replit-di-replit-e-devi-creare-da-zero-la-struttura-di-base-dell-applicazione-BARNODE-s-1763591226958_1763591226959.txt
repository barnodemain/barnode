6.1 ESLint e formattazione

6.2 Regole Anti-Monolite (file-length)

Crea uno script Node interno al progetto (es. scripts/check-file-length.mjs):

Scansiona file .ts e .tsx in src/.

Per ogni file:

Avvisa (log warning) se supera 200 righe.

Fallisce (exit code 1) se supera 300 righe.

Aggiungi script in package.json:

"check:filelength": "node scripts/check-file-length.mjs"

Integra nel pre-commit:

Usa Husky o il meccanismo standard di replit per i pre-commit.

Pipeline minima:

npm run lint

npm run check:filelength

7. BACKUP AUTOMATICI — COMANDO “esegui backup”

Prepara il meccanismo tecnico, senza dipendere da Supabase:

Crea una cartella backup/ nella root (può contenere un .gitkeep o essere vuota).

Crea uno script (scegli tu shell o Node, basta che funzioni in ambiente standard replit):

Es.: scripts/backup_barnode.sh oppure scripts/backup_barnode.mjs

Funzionalità:

Crea un archivio compresso nella cartella backup/ con nome tipo:

Backup_19 Nov_22.51.tar.gz (usa timestamp leggibile giorno/mese/ora_minuti, non serve millisecondo)

Esclude:

node_modules

dist o build

eventuali cartelle temporanee (es. tmp, .cache se presenti)

Mantiene solo gli ultimi 3 backup:

se ci sono più di 3 file in backup/, elimina quello più vecchio.

Configura il comando personalizzato replit:

Definisci un comando chiamato esattamente: esegui backup

Questo comando deve eseguire lo script di backup.

Se è richiesto un file di configurazione specifico (es. .replit, replit.config, tasks.json o simili), crealo/aggiorna nel modo più semplice possibile.

Nel report finale elenca:

percorso dello script

formato del nome file

logica di rotazione (come identifichi il più vecchio).
